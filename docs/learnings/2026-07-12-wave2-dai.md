## Headline

**Wave-2 adoption is a fundamentally different job than wave-1, and the prior's inverted.** The
first rollout's central fact was "a green repo lights up 1312 issues; expect 3/14 adoptable."
Here, **4 of 5 new checks were free** (0 issues) and the fifth was 25 issues. The reason is
structural, not luck: wave-1 checks encode *pervasive style conventions* (nil comparison, single-letter
vars, Logger shape) that a codebase either followed for years or didn't. Wave-2 checks encode
*specific anti-patterns* (`cast(attrs, Map.keys(attrs))`, `@derive Jason.Encoder` on a schema,
two modules in one file) that most codebases simply never commit.

**Prior for the next adopter:** narrow anti-pattern checks land free far more often than
convention checks. Don't budget wave-1-sized triage effort for a wave-2-shaped batch â€” but do

## Gotcha: `gh pr checks` served results for the *previous* head

The most dangerous thing that happened this cycle. Minutes after pushing `adff931`,
`gh pr checks 49 --watch` reported `Credo pass`, `Test & Coverage fail`, and **exited 0** â€” while
the runs for my actual SHA were still queued. Every line it printed belonged to `e0dd7b6`, the
prior head. Had I trusted it, I would have reported "Credo passes on my change" without a single
CI job having compiled my change.

The tell was run IDs: the "passing" runs were `292103475xx`, while the freshly-queued ones were
`292181144xx`. `--watch` did not wait, because from its view the checks it knew about were terminal.

**Rule: resolve every run against your own HEAD SHA before believing any verdict.**

```bash
git rev-parse HEAD
gh run list --repo <o>/<r> --branch <br> --limit 12 \
  --json databaseId,name,status,conclusion,headSha \
  --jq '.[] | select(.headSha|startswith("<sha7>")) | "\(.name) => \(.status)/\(.conclusion)"'
```

Then block on those specific IDs with `gh run watch <id> -i 20` (note: `gh run watch` has no
`--compact`; only `--exit-status` and `-i/--interval`). This generalizes the skill's existing
"match on failing-test identity, never shard index" caution one level earlier: **match on SHA
before you even get to the test names.** A stale green is more dangerous than a red.

## Deferral rationale worth reusing: the producer boundary, not the site count

`NoAtomStringKeyFallback` hit 25 sites across 10 files â€” reads like
`Map.get(ticket, :title) || Map.get(ticket, "title")` in the pipeline nodes and the reassessor.

Site count alone would frame this as "25 mechanical edits." It isn't, and articulating *why*
is the transferable part: **each read site is a symptom; the defect is upstream.** The `ticket`,
`thread`, and `check` maps genuinely arrive atom-keyed from internal factories/structs and
string-keyed from the Linear and GitHub JSON APIs. Deleting a fallback at a read site doesn't
normalize the map â€” it just picks one producer to break. The honest fix is to normalize at each
producer boundary, which means knowing every producer of every one of those maps: a real
refactor with real behavioral risk, not a rewrite.

This is the same shape as the skill's "semantics-preserving beats file count as the axis," but
sharpened into a question worth asking of any `||`-fallback or defensive-read check:

> Is this check flagging the *place where the bug shows up*, or the *place where the bug is*?

When those differ, defer â€” and say so in the deferral comment, because "25 issues, too much churn"
would have been a lie. The reason isn't churn, it's that the read sites aren't where the fix goes.

## Pattern that worked: enable-at-zero is the whole point of wave-2

Three checks were free. That is not a boring outcome â€” it's the deliverable. Each one now *locks in*
behaviour the repo already has (no mass-assignment `cast`, no schema-level `Jason.Encoder`, one
module per file) at zero diff cost. The temptation with a 0/0 check is to wonder whether it's worth
enabling; it is, and it's the cheapest guardrail available. The counterpart temptation â€” leaving a
0/0 check disabled "for now" â€” silently forfeits enforcement of a convention the repo currently and
freely satisfies.

Corollary on the instruction I was given ("NEVER fix SingleModulePerFile this pass, defer with
counts"): it came back 0/0, so there was nothing to fix and deferral would have been strictly worse
than enabling. **A pre-issued "defer this one" instruction is conditioned on the check being
non-zero.** Run the counts before honoring a deferral order.

## Anti-pattern avoided: trusting a zero without proving the check ran

Three of five checks reported 0. Per `verifying-credo-checks`, a 0 is exactly what a check that
never loaded, or a config whose globs match nothing, also reports. Cheap proof, done per check:

```bash
MIX_ENV=test mix credo --strict --only MikaCredoRules.<Check> 2>&1 | grep -E "running|found"
# => "0.02s running 1 check on 484 files" / "found no issues"
```

`484 files`, not `0 files`, is what converts "found no issues" from a claim into evidence. The
aggregate run corroborated it (`9 checks` before â†’ `13 checks` after), which also catches the case
where a check silently fails to load and vanishes from the count. Two independent signals, both free.

## Key decision: matched the file's deferral style over the literal instruction

Instruction said defer as `{Check, false}`. The repo's `.credo.exs` had seven prior deferrals, all
written as a commented-out entry under a `# not yet adopted: <counts> â€” <reason>` block. Introducing
`{Check, false}` would have left two spellings of "deferred" in one file, which defeats the greppable-
backlog property the convention exists for. Matched the existing style, flagged the deviation upward
rather than deciding silently.

**Generalization:** when a rollout instruction and the target file's established convention conflict
on *form* (not substance), the file wins, and you report the conflict. The instruction's *intent*
(quantified, greppable, in-place backlog) was fully satisfied either way â€” which is the test for
whether it's safe to deviate.

## CI exoneration, wave-2 flavour

`Test & Coverage` was red. Chain, in the order it should be built:

1. No path from the diff (`.credo.exs`, `mix.lock`, one `test/support` file in `developer_ai_service`)
   to `local_ticketing`, where every failure lived.
2. The only app I touched went **135 tests / 0 failures** locally and in CI.
3. Decisive: `main`'s own run (`29211451660`, sha `1d76061`) fails on the *identical* test module and
   test name â€” `LocalTicketing.Integration.LocalTicketingE2ETest` "writing a file to inbox/ ingests a
   ticketâ€¦", plus `WorkspacesWatcherTest`.

Detail worth keeping: the specific `WorkspacesWatcherTest` *cases* differed between the two runs while
the *module* was constant. That's the signature of a flake, and it's why the skill says match on
failing-test identity â€” but at **module** granularity when the check is flaky, since case-level
identity won't match run to run. Matching too tightly would have wrongly suggested a new failure.
