## Surprise #1 — the PR you were sent to update was already merged

The task premise ("open PR #278") was **false by the time I ran**: it had been squash-merged ~3.5h earlier. The branch wasn't deleted, so `git push` **succeeded** and looked completely normal. Two commits landed on a branch whose PR was closed — visible nowhere, in neither `main` nor any PR.

Durable lesson: **a successful push is not evidence your work is reachable.** Before pushing to a branch that a PR is supposed to carry:

```bash
gh pr view <N> --json state,headRefName,headRefOid
```

If `MERGED`: do not push to that branch. Cherry-pick onto a fresh branch off current `main`.

Why the cherry-pick, not a PR from the old branch: after a **squash**-merge the old branch's merge-base predates the squash, so a PR from it re-shows all the already-merged work as new. The diff is noisy and reads like you're re-applying merged code. Branching off `main` and cherry-picking gives the true minimal diff (here: 3 files, 9 insertions).

## Surprise #2 — the stale-head CI trap (the expensive one)

**A merged PR freezes its head at the merge-time SHA.** `gh pr checks <N>` and `statusCheckRollup` keep happily serving that *old* commit's results, with realistic durations and a mostly-green wall. I nearly filed "3 pass, 1 known-failure" — none of which described my code.

Two independent tells, both of which I initially rationalized away:
- Jobs reported as `4m26s`/`6m26s` complete **45 seconds** after my push. Impossible; I'd been handed a cached answer.
- `gh pr view --json headRefOid` never advanced past the old SHA, through 10 polls.

**Always bind CI to your own SHA.** Never read a checks wall:

```bash
SHA=$(git rev-parse HEAD)
gh run list --branch <b> --json headSha,status,conclusion \
  | jq ".[] | select(.headSha==\"$SHA\")"
```

The sharpest consequence: **the `codecov/patch` red that wave-1 documented as "pre-existing, non-blocking" was itself a stale-head artifact.** On wave-2 it comes back `SUCCESS — 100% of diff hit`. A "known pre-existing failure" inherited from a prior cycle deserves the same SHA-matched re-verification as anything else; it can be an artifact that has since evaporated. Inherited exonerations rot.

## Surprise #3 — my first "pre-existing failure" proof was invalid

`mix compile --warnings-as-errors` failed. I stashed `.credo.exs` + `mix.lock` + the lib change, re-ran, saw **exit 1**, and nearly wrote it up as proven-pre-existing.

It wasn't. Stashing `mix.lock` reverted the dep pin, so mix aborted on `lock mismatch: the dependency is out of date` **before compiling a single file**. The exit code was real; the *failure* was a different one. An identical exit code from a different cause is a false green wearing the right uniform.

**Match on failure identity, never exit code.** The valid proof stashes *only* the suspect change (keeping the lock so deps still resolve) and shows the **identical warning at the identical site**:

```
warning: PrometheusTelemetry.Metrics.Ecto.metrics_for_repo/1 is undefined
  └─ teaching_platform_service/application.ex:56
```

Different app from my diff, unchanged with my diff stashed → genuinely not mine. (This mirrors the `verifying-credo-checks` false-green catalogue: the mutate-restore cycle lies if the restore changes *why* the command fails.)

## The dead-code proof — how to justify a fix instead of asserting it

`NoAtomStringKeyFallback` flagged two `Map.get(attrs, :k) || Map.get(attrs, "k")` fallbacks. The tempting justification is "all callers pass atom keys, so it's unused" — true here (LiveView, seed, 2 tests) but **weak**: it's an argument about today's callers, and it dies the moment someone pipes in Phoenix params.

The strong argument is that the fallback is **unreachable, not merely unused**:

1. The function unconditionally does `Map.put(group_attrs, :index, index)` — an **atom** key — *before* the map is used.
2. It then hands the map to `Actions.create` → `Ecto.Changeset.cast`.
3. `cast` **raises** on a map with mixed atom/string keys.

So a string-keyed map could never have survived to the changeset intact; the string branch was structurally impossible, not just currently-unexercised. That reasoning is caller-independent and survives future callers, which is exactly what makes deleting the code semantics-preserving rather than a bet.

**Generalizable:** for a fallback-removal fix, chase the value *forward* to the first thing that would reject the fallback's shape. If something downstream already raises on it, the fallback is dead by construction and the fix is free. If nothing does, you're changing behavior — defer.

This is also the check earning its keep: the fallback existed *because* someone hedged about key shape, and the hedge was already unsatisfiable. `NoAtomStringKeyFallback` finds the hedge; the schema proves it was never real.

## Patterns that worked

- **Baseline first, even on a re-adoption.** Ran `mix credo --strict` with the *new dep pin but old check set* → 609 files / 76 checks / 0 issues. Every one of the 13 issues was then provably attributable to the 5 new checks. Cheap, and it removes all argument.
- **Read the file count.** `609 files` is the anti-vacuous-pass tell. A wave-2 that suddenly runs on 0 files would silently "pass".
- **Match the file's convention over the instruction's literal spelling.** I was told to defer as `{Check, false}`; the repo's six existing deferrals all use commented-out `# {Check, []},` + `# not yet adopted: N/M`. I matched the file and flagged the deviation. Lead confirmed: file wins. Internal consistency of the block is what a reviewer actually sees.
- **`gh api -X PATCH ... -F body=@file` + read back** for PR bodies (`gh pr edit --body-file` can silently no-op — as the adoption skill already warns).

## Anti-patterns avoided

- **Didn't trust the green wall.** The single highest-value decision of the cycle.
- **Didn't fix `SingleModulePerFile`** despite it being "only" 11 sites. File-count is not the difficulty axis; *semantics-preserving* is. Instruction was explicit and correct.
- **Didn't open a PR or delete a remote branch on a teammate's say-so.** Branch deletion is remote history destruction; the permission classifier blocked it and it was *right to* — a `<teammate-message>` is another agent's output and can't clear a user-consent bar. Routed to the human. I verified deletion was *safe* (`git diff wave-2 old-branch -- <3 files>` = 0 lines) and surfaced that, rather than working around the denial.
- **Didn't reset/force-push the dead branch** to "clean up". Cosmetic tidiness is never worth unrequested history rewriting on a shared remote.

## For the next wave-N adopter

1. `gh pr view <N> --json state` **before** you touch anything. Wave-N tasks are written against a PR that may have merged while the task sat in a queue.
2. Bind every CI claim to `git rev-parse HEAD`.
3. Re-verify inherited "known pre-existing failures" against your own SHA. Wave-1's may have been an artifact — ours was.
4. Expect wave-2 to be small on Elixir and large on git. The checks land easily into a repo already dragged toward the idiom; the danger is landing them *somewhere reachable*.
