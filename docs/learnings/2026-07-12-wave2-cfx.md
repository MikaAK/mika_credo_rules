## A zero can be *evidence the check is right*, not evidence it never ran

The skill says don't trust "found no issues". Correct — but the follow-up matters. Two independent confirmations, cheap, do different jobs:

1. **Non-vacuity** — the analysis line (`running 1 check on 1306 files`) proves the glob matched. A `0 files` here is the vacuous-green trap.
2. **Semantic cross-check** — grep for what the check hunts, then read the hits.

The second one paid off in a way I didn't expect. `NoAtomStringKeyFallback` returned 0, and grepping `Map.get(x, :a) || Map.get(...)` found two live sites:

- `apps/cfx_cms/lib/cfx_cms_web/components/canary_components.ex:608` — `Map.get(node, :name) || Map.get(node, :instance_id)`
- `apps/theta_data_api/lib/theta_data_api.ex:247` — `Map.get(eod, :timestamp) || Map.get(eod, :last_trade)`

Both are `||` fallbacks between **different key names** — precisely the case the check documents as *never flagged*. So the zero wasn't the check sleeping; it was the check being correctly narrow. **A grep that finds near-misses the check declined to flag is stronger evidence of correctness than a grep that finds nothing at all.**

## NoIdentityRewrap: the catch-all clause is the safety discriminator

The brief warned that removing an identity `case` "drops a shape assertion" — fix only the obviously-redundant. The useful sharpening: *whether it drops an assertion is mechanically decidable.*

**If the case has a catch-all clause (`error -> error`), it asserts nothing.** Anything unmatched falls through untouched, so there is no `CaseClauseError` to lose and removal is provably behavior-preserving. If instead every clause is a specific pattern with **no** catch-all, the case *does* assert shape — a surprise value would raise — and removal silently widens the accepted input. That's the line to check first, before any judgment call about "obviousness".

Second, subtler trap in the same check — **map re-wraps are not automatically identity:**

```elixir
{:ok, %{trades: trades, aggregate: agg}} -> {:ok, %{trades: trades, aggregate: agg}}
```

As a *pattern* `%{a: x, b: y}` matches maps with **extra keys**; as a *constructor* it builds a map with **exactly** those keys. So an "identity" re-wrap can silently **drop extra keys** — removing it would then change behavior. It's only safe once you've traced the producer and confirmed it already emits exactly that key set.

For `apps/cfx_web/lib/cfx_web/feed_table_data_adapters/trades_server_adapter.ex:17`, tracing `TradesServer.get_trades_data/1` → `CFXWeb.TradesCache.get_trades_data/1` (`trades_cache.ex:31-42`) showed the producer *itself* rebuilds `%{aggregate:, trades:}` with exactly two keys — so the adapter's re-wrap could not drop anything, and it had a catch-all. Provably safe. (Amusing corollary: `trades_cache.ex:36-37` is the same re-wrap shape but correctly *not* flagged, because its `{:ok, nil} -> {:error, ...}` clause does real work.)

The other site, `apps/options_chain_processor/.../option_chain_poll_scheduler.ex:121`, was `[] -> [] / contracts -> contracts` — a no-op on a list, no map subtlety.

## Decision rule: fix *all* of a check's sites, or defer it — never half

Partial fixes buy nothing. A check with 1 remaining issue is just as deferred as one with 2, so half-fixing is pure diff churn with no adoption payoff. The right question is never "is this site fixable?" but **"can I get this check to zero?"** If yes, fix every site in one commit and enable. If any single site is doubtful, defer the whole check and leave all sites alone. Both `NoIdentityRewrap` sites cleared, so it went in; had only one cleared, I'd have reverted both.

## `git stash` generalizes from the credo baseline to *any* red

Wave-1's headline technique was baseline-stashing `.credo.exs` to prove the issues were yours. The same move settles every "is this red mine?" question, and I used it three times:

```bash
git stash push <only-my-lib-files>   # keep config staged, drop the code change
<re-run the failing thing>            # identical failure = not yours. Done.
git stash pop
```

- 20/20 test failures in `options_chain_processor` → identical with my change stashed (local Redix AUTH).
- 2 `clause will never match` compile warnings → identical on the clean tree (`last_symbol_price_cache`, elixir_cache typing).

This is faster and far more decisive than reading logs and reasoning about causality. **Reach for it before you reach for the log.** Note it needs *path-scoped* stashing — stash only the lib files, or you drop the `.credo.exs` under test too.

Corollary worth internalizing: **this repo's test suite cannot run locally at all.** `cfx_web` setup migrations need the `timescaledb` Postgres extension. That is not fixable in a worktree and not worth trying; CI has it. Verify what you *can* locally (credo, compile, the stash-diff), state plainly what you couldn't, and let CI cover the rest.

## CI forensics: three ways `gh pr checks` lies

1. **It snapshots a moving target.** `codecov/project` reported **fail**, then settled to **pass** (74.09%, target 65%) once the remaining coverage uploads landed — the dead shard had caused a partial-upload transient. I nearly reported a false third red. If a coverage check is red, **re-read it after the run fully settles** (`gh api repos/{o}/{r}/commits/{sha}/check-runs` gives the current conclusion *and* `output.title`, which `gh pr checks` omits — the title is what reveals `74.09% (target 65.00%)`).
2. **Shard index is not failure identity.** The same coveralls failure moved from shard **3/3** (previous run) to **2/3** (mine). Matching on index would have read as "a new shard broke". Match on the failure *inside* the job.
3. **A red "Coverage" job may contain zero test failures.** Grepping the job log for `N tests, N failures` showed **every suite at 0 failures**; the job died on `##[error]Process completed with exit code 2` — the excoveralls **coverage threshold** (`0.0% apps/api_web/lib/api_web/error_json.ex`, a file untouched by the diff). "Coverage job red" ≠ "your tests broke". Establish which before spending any time on it.

Also confirmed the skill's advice: `gh api .../actions/jobs/<id>/logs | grep -a` is reliable where `gh run view --log-failed` is not.

## Stacking wave-N onto an open adoption PR

The wave-1 PR was still open, so wave-2 stacked directly onto `add-mika-credo-rules` — no new branch, no new PR. Keep the commit split the skill prescribes even when stacking: **fixes first (`df956ee6`), config-enable last (`c11db2c4`)**, so each commit is green in isolation.

Two conventions to match rather than invent, both read straight out of the existing `.credo.exs`:

- Enabled `MikaCredoRules` entries are **alphabetical**; deferred entries are **count-descending**. New deferrals slot by issue count (15 after 46, 9 after 10) — not appended.
- Deferral format is exactly `# not yet adopted: N issues / M files` + a one-line reason, then `{Check, false}`.

**Watch for the PR merging out from under you.** Sibling lanes (opgg #11, lex #278) were merged mid-task while their agents were working. Before pushing to a long-lived adoption branch, check `gh pr view <n> --json state,mergedAt`; if it merged, cherry-pick onto current `main` as a fresh branch instead of pushing to a dead one.
